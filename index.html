Booking Flow

How every ride works

A detailed look at the full journey from app open to destination arrival.

1
Phone Verification (OTP Login)
Enter your +91 phone number. A 6-digit OTP is sent via MSG91 SMS. Verify within the time window — no passwords, ever.
New users: Complete your profile (name + language preference). Returning users: Go straight to the home screen. Firebase Auth manages your session automatically.
2
Fare Estimation
Enter pickup and destination. Chalo queries Google Maps for the exact route distance, duration, and road polyline.
Formula: Base Fare + (Distance × ₹/km Rate) + Booking Fee × Surge Multiplier + 5% GST. Fallback: If Google Maps is unavailable, Haversine distance × 1.45 road factor is used. You always see the final price upfront.
3
Driver Assignment
Your ride is broadcast simultaneously to the top 5 nearest verified, online drivers within the service radius.
Drivers have 30 seconds to accept. If all decline, the next batch of 5 is notified. If the initial radius has no drivers, the search expands automatically. If truly no drivers are available, you're notified instantly.
4
Active Ride — OTP + Live Tracking
Your driver's location updates in real time via Firebase RTDB. An animated marker shows their exact position on the map.
OTP Start: The driver inputs a 4-digit OTP (shown on your screen) to start the ride — confirming you're the right passenger. SOS: At any point, tap the SOS button to send your GPS coordinates to emergency contacts and Chalo's safety team.
5
Payment & Rating
On arrival, pay by UPI (Razorpay SDK) or Cash. Then rate your driver from 1–5 stars.
UPI: Razorpay opens in-app. Payment verified via HMAC-SHA256 webhook. Cash: Skips directly to the rating screen. You can skip rating, but a prompt appears for 30 seconds and again on your next app open.

Engineering Principles

Why we built it
the way we did

Every architectural decision in Chalo exists for a reason — performance, reliability, or developer experience.

🔐
Firebase Custom Token Flow
Firebase Phone Auth locks OTP delivery to Firebase's infrastructure. The custom token flow lets our backend control OTP generation and delivery via MSG91 — giving us reliability, cost control, and language support. The backend mints a custom token after OTP verification; Firebase SDK then manages the ID token lifecycle identically.
🗺️
PostgreSQL + PostGIS
PostgreSQL is the safe default for a transactional ride system. PostGIS adds full geospatial capability — ST_DWithin with GiST indexes — without sacrificing ACID guarantees. Driver search is accurate spherical distance, not an application-layer bounding box.
⚡
BullMQ Over cron / setInterval
setInterval doesn't survive server restarts and breaks under horizontal scaling. BullMQ persists jobs in Redis, deduplicates repeating jobs by name, and supports delayed one-shot jobs like ride-offer-expired. OpenTelemetry trace context propagates through async job boundaries.
🔄
Three-Tier Config Cache
Platform config is read on every ride request. L1 in-process cache is sub-millisecond. L2 Redis cache provides cross-instance consistency within 60 seconds. L3 PostgreSQL is the admin-controlled source of truth. Fares can be changed with zero deployment.
🔌
Pluggable KYC Provider
The IKYCProvider interface lets the system run with ManualKYCProvider in early ops and switch to SurepassKYCProvider by setting one environment variable — no code change, no migration needed.
📱
kotlinx.serialization Over Gson
Gson uses reflection for JSON mapping. R8 minification renames Kotlin property names in release builds — silently breaking deserialization. kotlinx.serialization generates code at compile time via KSP. All 5 DTO files were migrated from @SerializedName to @SerialName.
📋
Principle: Every state transition is audit-logged
RideEvent records every lifecycle change with timestamp, GPS, and metadata JSON. This creates an immutable audit trail for dispute resolution, driver penalisation review, and operational debugging — without touching the Ride table for history queries.
